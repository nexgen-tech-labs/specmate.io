# On-Prem Agent Security Model (Issue 11.3)

Design-only spec — no code in this issue. Defines the security properties the on-prem agent (Epic 11) must satisfy before it's built, written to be usable directly in a security review conversation with a prospective on-prem customer without rewriting. Companion to [`agent-relay-protocol.md`](agent-relay-protocol.md) (Issue 11.2), which defines the message contract this document constrains.

## Threat model summary (for a customer's security reviewer)

The agent is a process the customer runs inside their own network. It exists solely to let SpecMate's cloud publish items into an on-prem Jira/ADO/GitHub Server instance that SpecMate's cloud cannot otherwise reach. The customer is trusting SpecMate with: (a) no ability to reach anything else on their network, (b) no ability to read their on-prem tool's credentials, and (c) an auditable, revocable, minimum-privilege channel that only does what this document says it does.

## 1. Outbound-only connection model

**The agent never listens on any port.** All connections are initiated from inside the customer's network, outward to SpecMate cloud. This is the single most important property for a customer's security team: **no inbound firewall rule, no port forward, no DMZ placement is ever required.** The agent behaves like any other outbound HTTPS client on the customer's network — indistinguishable, from the network's perspective, from a browser or a `curl` call to an external API.

Concretely (per the transport recommendation in `agent-relay-protocol.md`): the agent long-polls a SpecMate cloud endpoint (`GET /agents/{id}/next-request`) and posts status callbacks back (`POST /agents/{id}/callback`) — both outbound, both plain HTTPS, both indistinguishable from normal egress traffic a firewall already permits for any SaaS integration. If a future revision moves to WebSocket or a queue, the same constraint holds: the agent always dials out, cloud never dials in.

**Explicit non-goal**: SpecMate cloud never needs, requests, or should be granted any inbound network path into the customer's environment. Any future design that requires an inbound rule is a spec violation and should be rejected at design review, not just at implementation time.

## 2. Encrypted tunnel requirements

- **TLS 1.2+ is the floor**, matching this repo's existing baseline (all Azure Container Apps ingress is TLS-terminated; no plaintext HTTP anywhere in the stack today). The agent's connection to SpecMate cloud is a standard HTTPS client connection — no new requirement beyond what already exists for every other client of the API.
- **Mutual TLS is the target, not just "considered."** Server-side TLS alone proves the agent is talking to the real SpecMate cloud; it does _not_ prove SpecMate cloud is talking to the real, registered agent for that workspace. The [11.2 auth handshake](agent-relay-protocol.md#1-auth-handshake) already requires every request to be signed with the agent's private key — mTLS is the natural way to enforce that at the transport layer rather than re-implementing request signing in application code. Recommendation: the agent's registration step (11.2) generates a client certificate (not just a signing keypair) during bootstrap, and SpecMate cloud's ingress is configured to require and verify that client cert for the `/agents/*` endpoint family specifically — while the rest of the API stays on standard server-only TLS. This is a meaningfully larger infra lift than the rest of Epic 11 (Azure Container Apps' native ingress needs a client-cert-verification layer, likely via a sidecar or an API-gateway in front of just this endpoint family) and should be scoped explicitly as its own task when Epic 11 is built, not assumed to fall out of "just use TLS."
- **Certificate pinning is explicitly not required.** Standard CA-validated TLS is sufficient; pinning adds operational fragility (cert rotation becomes a coordinated two-sided event) without a threat it meaningfully defends against here, given mTLS already authenticates both directions.

## 3. Credential isolation

This is the property most likely to be scrutinized by a security-conscious buyer, so it's stated as a set of hard invariants:

1. **The agent holds exactly one credential class: the on-prem tool's own credential** (Jira Server PAT, ADO Server PAT/basic-auth, GHES PAT — whatever `JiraConnection`/`AdoConnection`/`GitHubConnection` already model for cloud). This credential is configured locally on the agent (config file or env var, customer-managed, customer-rotated) and **is never transmitted to SpecMate cloud in any form** — not at registration, not in telemetry, not in error messages. The [item-push message](agent-relay-protocol.md#2-item-push-format-cloud--agent-please-publish-this) sent from cloud to agent carries `auth_hint: null` specifically to make this invariant visible in the wire format itself, not just in a policy document.
2. **Scoped per workspace, per publish mapping — never broader.** The `Agent` row (11.2's data model) binds one agent registration to exactly one `(workspaceId, publishMappingId)` pair. An agent registered for Workspace A's Jira Server mapping cannot be reused, even accidentally, for Workspace B or for a different tool — there is no "org-wide agent" concept. If a customer needs to relay to two on-prem tools (say, both Jira Server and ADO Server), that's two separate agent registrations, two separate keypairs, two separate revocation switches. This bounds the blast radius of a single compromised agent process to exactly one tool integration.
3. **The agent cannot reach SpecMate cloud's own infrastructure beyond the relay protocol.** The agent's only interaction with SpecMate cloud is the long-poll/callback endpoint pair defined in 11.2 — it has no database access, no access to any other API route, no credential that would let it authenticate as a SpecMate user or admin. From SpecMate cloud's side, the agent's signed requests are validated against a narrow allowlist of exactly two routes; a compromised agent gains nothing beyond "can fetch its own queued relay requests and post callbacks for them" — it cannot enumerate other workspaces' data, other agents' requests, or any unrelated SpecMate API surface.
4. **No standing credential SpecMate cloud can use to reach into the customer's network.** This is the inverse of invariant 1 and is worth stating explicitly for a reviewer: SpecMate cloud holds only the agent's _public_ key (for verifying signed requests) and metadata (`lastSeenAt`, which mapping it serves). There is no secret on the cloud side whose compromise would let an attacker reach the customer's on-prem tool — the private key and the on-prem tool's own credential both live exclusively inside the customer's network.
5. **Revocation is instant and cloud-side.** An admin revokes an `Agent` row from the SpecMate UI; every subsequent signed request from that agent is rejected at the next request (no waiting for a token to expire, no coordination needed with the agent process itself — the agent doesn't even need to be reachable to revoke its access). The customer does not need to trust SpecMate to "eventually" honor a revocation; the check is a stored `revokedAt` timestamp evaluated on every request.

## 4. What a customer's security team should review before installing the agent

A checklist, intended to be handed to a prospective on-prem customer's security reviewer as-is:

- [ ] Confirm the agent binary/container makes **no inbound-listening sockets** — verify via `netstat`/`ss` after starting it, or a network policy that denies inbound to the agent's host entirely (the agent should function identically with such a policy in place).
- [ ] Confirm all outbound traffic from the agent goes to SpecMate cloud's documented endpoint(s) only — an allowlisted egress rule to those hosts/IPs, denying all other outbound from the agent's host, should not break functionality.
- [ ] Confirm the on-prem tool credential configured on the agent has the **minimum Jira/ADO/GHES permission scope** needed to create/update/comment on issues in the specific project(s) being published to — not a global admin token. (This is a customer-side configuration choice, not something SpecMate enforces technically, but should be called out in the reviewer's checklist since it's the actual blast-radius control if the agent host itself is compromised.)
- [ ] Review the agent's registration/credential-bootstrap flow (11.2 §1) and confirm the one-time registration token is treated with the same handling rigor as any other bootstrap secret (short-lived, single-use, transmitted out-of-band or via the authenticated SpecMate web session only).
- [ ] Confirm log output from the agent (stdout/stderr, or wherever it's shipped) never contains the on-prem tool credential or the agent's private key — this should be a stated logging invariant in the eventual Go implementation, verifiable by grepping agent logs after a test run.
- [ ] Confirm the revocation flow (§3.5) actually cuts off access without requiring the agent process to be stopped or reachable — testable by revoking an `Agent` row and confirming the next relay request from that (still-running) agent is rejected.
- [ ] Review data flow: confirm that only the fields in the [item-push format](agent-relay-protocol.md#2-item-push-format-cloud--agent-please-publish-this) (method/url/headers/body needed for the specific Jira/ADO/GHES API call) transit the relay — not, e.g., the full `DraftItem` history, other workspace data, or anything beyond what's needed for this one publish call.
